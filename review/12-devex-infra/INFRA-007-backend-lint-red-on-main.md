## Problem
`uv run ruff check .` fails on main with 5 errors (UP035, I001, UP007 ×3) and `ruff format --check` flags the same file — all in the newest migration, `1307fd3f3b27_add_single_model_to_providers.py`. The CI lint job runs exactly these commands, so CI is (or will be on the next push) red for everyone.

## Evidence
Executed during this review (see `00-meta/environment.md`):
```
$ uv run ruff check .
UP035 [*] Import from `collections.abc` instead: `Sequence`
  --> alembic/versions/1307fd3f3b27_add_single_model_to_providers.py:8:1
I001 [*] Import block is un-sorted or un-formatted  (same file:8)
UP007 [*] Use `X | Y` for type annotations          (same file:16,17,18)
Found 5 errors.  [*] 5 fixable with the `--fix` option.
$ uv run ruff format --check .
Would reformat: alembic/versions/1307fd3f3b37_add_single_model_to_providers.py
1 file would be reformatted, 195 files already formatted
```
CI runs the same commands: `.github/workflows/ci.yml:31-33`.

## Impact
Every contributor's `make lint` fails on a clean checkout; PR CI fails for unrelated changes; the habit of skipping lint locally gets normalized.

## Root cause
The migration (commit `ce2ea6c`/`a311bd5` family) was generated from the default alembic template (old-style typing) and committed without running lint.

## Proposed fix (specification)
`cd backend && uv run ruff check --fix . && uv run ruff format .` — commit the result (one file). Optionally update `alembic/script.py.mako` to emit ruff-clean revision files (modern `|` unions, sorted imports) so future autogenerated migrations don't regress.

## Alternatives considered
Per-file ignore for alembic/versions — defensible (generated code), but the mako-template fix achieves cleanliness without exclusions.

## Verification
`uv run ruff check . && uv run ruff format --check .` exit 0; CI lint job green.
